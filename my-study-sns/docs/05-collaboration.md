## 5.3 협업 도구 및 방법론

### 버전 관리: Git / GitHub

#### 브랜치 전략

```
main (배포용)
  │
  ├── develop (개발 통합)
  │     │
  │     ├── feature/ai-note-analysis (AI 노트 분석)
  │     ├── feature/schedule-parser (시간표 분석)
  │     ├── feature/knowledge-cards (지식카드)
  │     ├── feature/auth-system (인증 시스템)
  │     ├── feature/timeline-ui (타임라인 UI)
  │     └── fix/bug-name (버그 수정)
  │
  └── hotfix/* (긴급 수정)
```

#### 커밋 컨벤션

| 접두사 | 용도 | 예시 |
|--------|------|------|
| `feat` | 새 기능 | `feat: AI 스터디 노트 자동 작성 기능 구현` |
| `fix` | 버그 수정 | `fix: AuthGuard 의존성 배열 정리` |
| `docs` | 문서 수정 | `docs: README 업데이트` |
| `style` | 코드 스타일 | `style: 코드 포맷팅` |
| `refactor` | 리팩토링 | `refactor: API 라우트 구조 개선` |
| `test` | 테스트 | `test: 이미지 업로드 테스트 추가` |

#### 실제 커밋 히스토리

```
ae9d8fc Fix: KaTeX CSS import를 layout.tsx로 이동
d4a301f Add: AI 스터디 노트 자동 작성 기능 구현
90a329f Add: 시간표 이미지 AI 분석 기능 구현
6529f5e Add: EC2 자동 배포 job 추가
461be9f Fix: AuthGuard 의존성 배열 정리 및 ImageScheduleForm 마크다운 수정
```

## 5.5 협업 사례

### 사례 1: AI OCR 시스템 개발

**상황**: 2단계 OCR 시스템 설계 및 구현

**협업 과정**:
1. **팀원 1**: API 설계 및 Gemini 연동 담당
2. **팀원 2**: 업로드 UI 및 진행 상태 표시 담당
3. **팀원 3**: 이미지 저장 및 데이터베이스 연동 담당
4. **팀원 4**: 다양한 이미지 형식/품질 테스트 담당

**결과**: 2주 내 핵심 기능 완성

### 사례 2: 시간표 분석 기능 개발

**상황**: 다양한 시간표 형식 지원 필요

**협업 과정**:
1. **팀원 4**: 다양한 시간표 샘플 수집
2. **팀원 1**: AI 프롬프트 설계 및 형식 감지 로직 구현
3. **팀원 2**: 시간표 미리보기/편집 UI 구현
4. **팀원 3**: 기존 시간표와의 충돌 검사 로직 구현

**결과**: 24시간제/12시간제/교시제 모두 지원

### 사례 3: 버그 수정 협업

**상황**: AuthGuard 의존성 문제로 무한 리렌더링 발생

**협업 과정**:
1. **팀원 4**: 버그 발견 및 재현 조건 정리
2. **팀원 3**: 원인 분석 (useEffect 의존성 배열 문제)
3. **팀원 1**: 수정 코드 작성 및 PR 생성
4. **팀원 2**: 코드 리뷰 및 UI 영향 검토

**결과**: 1일 내 수정 완료

## 5.6 갈등 해결 사례

### 기술 스택 선정 논의

**이슈**: 데이터베이스 선택 (Firebase vs Supabase)

**논의 과정**:
| 팀원 | 의견 |
|------|------|
| 팀원 1 | Supabase - PostgreSQL 기반으로 복잡한 쿼리 가능 |
| 팀원 2 | Firebase - 익숙하고 문서가 풍부함 |
| 팀원 3 | Supabase - RLS로 보안 관리 용이 |

**결정 방법**:
1. 각자 장단점 정리하여 공유
2. 프로젝트 요구사항과 비교 분석
3. 투표 진행 (3:1로 Supabase 선택)

**결과**: Supabase 채택, RLS를 활용한 보안 강화
